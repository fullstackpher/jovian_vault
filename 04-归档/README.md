# 04-归档 (Archive)

> 已完成或暂停的项目、过期不再维护的内容

## 📍 定位

归档文件夹用于存放**已完成、暂停或不再活跃维护**的内容。将完成的项目、过时的资料移到这里，可以保持工作区的简洁，同时保留历史备查阅。

**记录以核心原则**：定期归档，保留历史，保持工作区整洁。

## 📂 存储内容类型

- **已完成项目**：项目已结项，产出的文档和代码
- **暂停项目**：因各种原因暂停，暂时搁置的项目
- **过时知识**：领域中被新知识替代的旧内容
- **过期资源**：工具旧版本教程、过时的参考资料
- **历史版本**：重要文件的旧版本

## 📏 命名规范

### 归档文件夹命名

```
[原文件夹名称]-[YYYY-MM]/
```

### 归档文件命名

```
[原文件名]-[归档日期].md
```

### 示例

```
04-归档/
├── 01-项目-2026-01/              # 2026年1月归档的项目
│   ├── 01-学习Java-2026-01.md
│   └── 02-知识管理-2026-01.md
├── 02-领域-2025-12/              # 2025年12月归档的领域内容
│   └── 旧知识体系-2025-12.md
└── 03-资源-2026-01/              # 2026年1月归档的资源
    └── 过期教程-2026-01.md
```

### 命名要求

| 规则 | 说明 |
|------|------|
| 归档前缀 | 保留原文件夹编号 `01-`, `02-` 等 |
| 日期标注 | 使用 `[YYYY-MM]` 格式标注归档时间 |
| 保留原名 | 文件名尽量保留原始名称 |
| 禁止字符 | 不使用 `\/:*?"<>|` |

### 归档时间标注

| 类型 | 标注格式 | 示例 |
|------|----------|------|
| 按月归档 | `[YYYY-MM]` | `01-学习Java-[2026-01]` |
| 按季度归档 | `[YYYY-QN]` | `01-学习Java-[2026-Q1]` |
| 按年归档 | `[YYYY]` | `01-学习Java-[2026]` |

## 📁 推荐归档结构

```
04-归档/
├── 01-项目/                      # 已完成/暂停的项目
│   └── [YYYY-MM]/
│       ├── [原项目名]-[YYYY-MM].md
│       └── ...
├── 02-领域/                      # 过时的领域内容
│   └── [YYYY-MM]/
│       ├── [原文件名]-[YYYY-MM].md
│       └── ...
├── 03-资源/                      # 过期的资源
│   └── [YYYY-MM]/
│       ├── [原文件名]-[YYYY-MM].md
│       └── ...
└── README.md                     # 本文件
```

## 🔄 归档时机

| 触发条件 | 操作 |
|----------|------|
| 项目完成 | 项目结项后 1 周内归档 |
| 项目暂停 | 暂停决定确定后立即归档 |
| 内容过时 | 发现内容已过时时归档 |
| 定期清理 | 每月检查一次，主动归档 |

## ⚠️ 注意事项

1. **不要**删除归档内容，只能移动
2. **不要**在归档文件夹中创建新内容
3. **保持**归档结构清晰，便于日后查找
4. **标记**归档日期，方便追溯时间线

## 💡 归档 vs 删除

| 归档 | 删除 |
|------|------|
| 未来可能需要参考 | 确认永不需要 |
| 项目历史记录 | 临时文件/草稿 |
| 有价值的旧版本 | 重复内容/过时无价值 |
| 需要保留上下文 | 内容完全无意义 |

## 📊 归档统计

定期检查归档内容，评估是否需要：

- **保留**：仍有参考价值
- **压缩**：合并相似内容减少空间
- **清理**：彻底删除（谨慎操作）

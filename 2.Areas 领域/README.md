# 2.Area 领域 (Areas)

> 长期追踪的领域、持续深耕的知识体系、技能提升计划

## 📍 定位

领域文件夹用于存放**需要长期积累和持续深耕**的知识体系。与项目不同，领域没有明确的截止日期，是需要一辈子持续学习和维护的方向。这是构建个人专业深度的地方。

**核心原则**：深度积累，连接整合，迭代更新。

## 📂 存储内容类型

- **知识体系**：某一领域的完整知识结构和概念网络
- **技能树**：需要掌握技能的分解和学习路径
- **概念卡片**：领域内的核心概念、术语定义
- **最佳实践**：总结提炼的方法论和经验
- **学习路径**：系统学习该领域的路线指南
- **深度笔记**：对重要主题的深入研究和分析
- **案例研究**：典型案例、解题思路、模式总结

## 📏 命名规范

### 领域文件夹命名

```
[领域英文名称]/
```

### 领域内文件命名

```
[领域英文名称]/[主题]-[描述].md
```

### 示例

```
Backend/
├── 01-基础知识体系.md
├── 02-核心概念卡片/
│   ├── 数据库设计.md
│   └── API设计原则.md
├── 03-技术栈学习/
│   ├── Spring框架.md
│   └── MyBatis深入.md
└── 04-最佳实践/

Frontend/
├── 01-前端架构.md
├── 02-React深入.md
└── 03-性能优化.md

Knowledge Management/
├── 01-知识管理方法论.md
├── 02-Obsidian使用技巧.md
└── 03-双链笔记实践.md
```

### 命名要求

| 规则 | 说明 |
|------|------|
| 文件夹名称 | 使用 **英文** 驼峰命名（如 `Backend`, `Frontend`） |
| 文件前缀 | 使用两位数字 `01-`, `02-` 等排序 |
| 主题分类 | 清晰分类，保持逻辑层次 |
| 禁止字符 | 不使用 `\/:*?"<>|` 和空格 |

### 文件前缀对照表

| 前缀 | 类型 | 说明 |
|------|------|------|
| `01-` | 基础知识体系 | 该领域的核心知识框架 |
| `02-` | 概念卡片 | 核心概念、术语定义 |
| `03-` | 技术栈学习 | 工具/框架/库的学习记录 |
| `04-` | 最佳实践 | 方法论、经验总结 |
| `05-` | 案例研究 | 典型案例分析 |
| `06-` | 学习路线 | 系统学习路径 |
| `07-` | 进阶主题 | 高级话题深入研究 |

## 🏗️ 领域结构模板

```
[领域英文名称]/
├── README.md                    # 领域介绍和学习路线
├── 01-基础知识体系.md           # 核心知识框架
├── 02-概念卡片/                 # 概念定义卡片
│   ├── 概念A.md
│   └── 概念B.md
├── 03-技术栈学习/               # 工具/框架学习
│   ├── 技术A.md
│   └── 技术B.md
├── 04-最佳实践/                 # 方法论总结
├── 05-案例研究/                 # 案例分析
└── 06-进阶主题/                 # 高级内容
```

## 🔄 维护周期

| 周期 | 动作 |
|------|------|
| 每日 | 遇到相关新知识时添加到对应领域 |
| 每周 | 复习和更新领域内容 |
| 每月 | 审视领域结构，补充缺失内容 |
| 每季 | 评估学习进度，调整学习路径 |

## 💡 与项目的关系

- **项目** 产出 **领域知识**（在项目中学习到的提炼到领域）
- **领域知识** 指导 **项目执行**（用领域积累来做好项目）
- 两者相互促进，形成正向循环

## ⚠️ 注意事项

1. **不要**将所有内容堆在一个文件中，保持结构清晰
2. **不要**追求数量，注重质量和深度
3. **保持**概念卡片之间相互链接，形成知识网络
4. **定期**审视领域结构，删除过时内容
